# CLAUDE.md — Next.js 15 + SQLite SaaS Project

## Stack & Versions

- **Framework:** Next.js 15 (App Router), React 19
- **Database:** SQLite via `better-sqlite3` (local) or Turso (production edge)
- **ORM / Query:** `drizzle-orm` with drizzle-kit for migrations
- **Auth:** NextAuth.js v5 (Auth.js) with credentials + OAuth providers
- **CSS:** Tailwind CSS v4 with `tailwind-merge` + `clsx` for class composition
- **Validation:** Zod for runtime validation (forms, API inputs, env vars)
- **TypeScript:** Strict mode, `@total-typescript/ts-reset` for better array/tuple types
- **Package Manager:** pnpm (because npm has hoisting bugs and yarn is slow)

Do NOT add: Redux, Zustand, React Query, tRPC, or any state/API client unless explicitly asked. Server Actions + React Server Components cover 95% of use cases in this stack.

## SQL / Migration Conventions

1. **NEVER write raw SQL** — always use Drizzle ORM or Drizzle schema definitions.
2. **Migrations are auto-generated:** `pnpm db:generate` → `pnpm db:migrate`.
3. **Every table must have:**
   - `id: integer().primaryKey({ autoIncrement: true })` (or cuid/uuid if distributed)
   - `createdAt: text().notNull().default(sql\`(datetime('now'))\`)`
   - `updatedAt: text().notNull().default(sql\`(datetime('now'))\`).$onUpdate(() => sql\`datetime('now')\`)`
4. **Foreign keys:** Define them in Drizzle via `references()`. SQLite needs `PRAGMA foreign_keys = ON` at connection time (done in `db.ts`).
5. **Indexes:** Add indexes for columns used in `WHERE`, `ORDER BY`, or `JOIN`. Name them `idx_<table>_<column>`.
6. **NO `SELECT *`** — always specify columns. This prevents surprises when the schema evolves.
7. **Soft deletes** are preferred over hard deletes. Add `deletedAt: text()` to tables that might need recovery.

## Component Patterns

1. **Server Components by default.** Only add `'use client'` when you need:
   - Event handlers (onClick, onSubmit)
   - Browser-only APIs (localStorage, IntersectionObserver)
   - useState / useEffect / useReducer
   - Context providers
2. **Server Actions** handle mutations. Call them from forms with `useActionState`.
3. **UI components** (Button, Input, Card) accept `className` and merge via `cn()` from `@/lib/utils`.
4. **Forms:** Use Server Actions + HTML form attributes. Avoid React Hook Form / Formik unless the form has 10+ fields with complex validation.
5. **Loading states:** Use `loading.tsx` and `Suspense` boundaries. NO loading spinners in Server Components — use streaming SSR instead.

## Dev Commands

```bash
pnpm dev            # Start dev server
pnpm build          # Production build
pnpm db:generate    # Generate Drizzle migrations
pnpm db:migrate     # Apply migrations
pnpm db:seed        # Seed the database
pnpm lint           # ESLint + Prettier
pnpm test           # Vitest
pnpm typecheck      # tsc --noEmit
```

## What We Don't Do (And Why)

| Anti-pattern | Reason |
|---|---|
| `rm -rf node_modules && pnpm install` as a "fix" | Corrupts the lockfile; use `pnpm store prune` instead |
| Barrel files (re-exporting everything from `index.ts`) | Breaks tree-shaking in Next.js; import directly |
| Default exports in components | Named exports give better IDE autocompletion and refactoring |
| Client Components for static UI | Increases bundle size; Server Components stream HTML for free |
| Raw SQL queries | Drizzle handles type safety; raw SQL bypasses migration tracking |
| Next.js Pages Router | App Router is the future; Pages Router is in maintenance mode |
| `any` type in new code | If it's truly unknown, use `unknown` with type guards |

## Enforcement Notes

- ESLint rule: `@typescript-eslint/no-explicit-any: error` (use `unknown`)
- ESLint rule: `import/no-default-export` for non-page files
- Prettier: `trailingComma: "all"`, `semi: true`, `singleQuote: true`
- `tsconfig.json` has `strict: true`, `noUncheckedIndexedAccess: true`, `exactOptionalPropertyTypes: false` (SQLite doesn't distinguish null from undefined)

## Testing Patterns

- **Unit tests:** Vitest + `@testing-library/react` for component tests
- **DB tests:** Use an in-memory SQLite database (`:memory:`) — created per-test via `beforeEach` (schema + seed)
- **E2E tests:** Playwright (with `@playwright/test`) — run against the preview deployment
- **Mocking:** Avoid mocking the DB. Use in-memory SQLite. It's fast enough for hundreds of tests.
